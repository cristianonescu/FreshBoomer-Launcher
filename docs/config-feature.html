1. Architecture Overview

FreshBoomer uses a centralized, file-based configuration system built around two core components:

ConfigData

An immutable Kotlin data class holding every configurable value in the app — from the user's nickname to feature toggles to TTS speech rate.

AppConfig

A Kotlin object (singleton) that owns the configuration state, persists it to a JSON file, and exposes it reactively via StateFlow.

The system also supports legacy SharedPreferences sync (for backward compatibility), remote JSON import & auto-sync (fetch config from a URL, with automatic re-fetch on every main screen resume), a setup wizard (first-run experience), and a JSON config editor for advanced users. The remote config URL is stored independently in SharedPreferences (via ConfigUrlPreference) so that remote JSON payloads can never overwrite or clear it.

Key principle: All configuration changes flow through AppConfig.save(), which updates the in-memory StateFlow, writes to disk, syncs to SharedPreferences, and merges quick contacts into the QuickContactRepository. This ensures a single source of truth.

2. ConfigData Model

app/src/main/java/ro/softwarechef/freshboomer/data/ConfigData.kt

ConfigData is an immutable data class with sensible defaults for every field. It groups 20+ fields into logical categories:

General Settings

FieldTypeDefaultDescription

themeMode Enum "SYSTEM" App theme. Options: SYSTEM, LIGHT, DARK
appLanguage Enum "ro" App language (currently only Romanian)

Text-to-Speech

FieldTypeDefaultDescription

ttsEngine Enum "PIPER_LILI" Voice engine. Options: PIPER_LILI, DEVICE_DEFAULT
ttsSpeechRate Float 0.85 Speech rate: 0.5 (slow) to 2.0 (fast)

Feature Toggles

FieldTypeDefaultDescription
featureQuickContacts Boolean true Show quick-dial buttons on home screen
featureDialPad Boolean true Show dial pad button
featureContacts Boolean true Show contacts button
featureMessages Boolean true Show SMS messages button
featureGallery Boolean true Show photo gallery button
featureWhatsapp Boolean false Show WhatsApp button on home screen

Behavior Settings

FieldTypeDefaultDescription
autoMaxVolume Boolean true Auto-set all audio streams to max volume
inactivityTimeoutMs Long 20000 Milliseconds before auto-return to home screen
maxMissedCallAnnouncements Int 3 Max times a missed call is announced via TTS
callSpeakerDelayMs Long 3000 Delay before auto-enabling speakerphone
inactivityMonitorEnabled Boolean false Send SMS alerts to emergency contacts on prolonged inactivity
inactivityMonitorThresholdHours Int 12 Hours of inactivity before SMS alert is sent

Contact Lists

FieldTypeDefaultDescription
quickContacts List<QuickContact> [] (empty) Quick-dial contacts shown as large buttons on home screen
emergencyContacts List<EmergencyContact> [] (empty) Contacts notified via SMS during inactivity alerts

Serialization

ConfigData has two methods for JSON conversion:

  • toJson(): JSONObject — serializes all fields to a JSONObject, stored with 2-space indentation
  • fromJson(json: JSONObject): ConfigData (companion) — deserializes with graceful fallback to defaults via optString, optBoolean, etc.
Resilience: Every field in fromJson() uses opt* methods with defaults. This means a partial JSON (e.g., from a remote URL) will fill in missing fields with defaults rather than crashing.

3. AppConfig Singleton

app/src/main/java/ro/softwarechef/freshboomer/data/AppConfig.kt

AppConfig is a Kotlin object (process-wide singleton) that acts as the central state manager for all configuration. It owns the StateFlow and coordinates all persistence.

Public API

MemberSignatureDescription
configFlow StateFlow<ConfigData> Observable stream of the current config. Compose UI and other observers collect this.
current ConfigData (getter) Synchronous snapshot of the current config value.
init(context) @Synchronized Loads config from config.json, or migrates from SharedPreferences on first launch.
save(context, config) @Synchronized Persists a new config: updates flow → writes file → syncs SharedPreferences → syncs contacts repo.
resetToDefaults(context) @Synchronized Replaces current config with a fresh ConfigData() (all defaults).
importFromUrl(context, url) Result<Unit> Fetches JSON from a URL and applies it. Must be called from a background thread.

Internal Methods

MethodWhat it does
writeFile() Writes config.toJson().toString(2) to config.json in the app's internal files directory.
migrateFromSharedPrefs() One-time migration: reads old LauncherPrefs SharedPreferences, converts to ConfigData, and writes the JSON file. Preserves existing quick contacts from QuickContactRepository.
syncToSharedPrefs() Writes key config fields back to SharedPreferences for backward compatibility with legacy code paths.
syncContactsToRepo() Merges quick contacts from config into QuickContactRepository, preserving photo URIs and drawable resource names from existing contacts.
Thread safety: All public mutating methods (init, save, resetToDefaults) are @Synchronized. The importFromUrl method performs network I/O and must be called from a coroutine or background thread.

4. Lifecycle & Initialization

App Startup Sequence

App process starts → MainActivity.onCreate() → AppConfig.init(context)
↓
config.json exists?
YES ↓              NO ↓
Parse JSON → ConfigData    migrateFromSharedPrefs()
↓                             ↓
_configFlow.value = config
↓
syncToSharedPrefs() → Ready!

Step-by-step:

  1. MainActivity.onCreate() calls AppConfig.init(this) as the very first action.
  2. init() checks if config.json exists in the app's internal files directory.
  3. If it exists: reads the file, parses JSON into ConfigData via ConfigData.fromJson(), and sets the flow value. On parse failure, falls back to defaults and rewrites the file.
  4. If it doesn't exist (first launch): calls migrateFromSharedPrefs() to pull values from legacy LauncherPrefs, creates a ConfigData, and writes the new JSON file.
  5. In both cases, syncToSharedPrefs() runs to keep the legacy preference store in sync.
  6. After init, QuickContactRepository.ensureDefaults() and InactivityMonitorWorker.schedule() run, both depending on the now-loaded config.

Boot Receiver

BootReceiver also calls AppConfig.init(context) on device boot to ensure config is loaded before the inactivity monitor worker is scheduled.

5. Storage Layers

Configuration is persisted in multiple locations for reliability and backward compatibility:

Primary: config.json

context.filesDir/config.json

The source of truth. A pretty-printed JSON file (2-space indent) containing all ConfigData fields. Read on startup, written on every save.

Secondary: SharedPreferences

LauncherPrefs (MODE_PRIVATE)

Kept in sync by syncToSharedPrefs() for legacy code that reads preferences directly. Contains a subset of fields (nickname, theme, TTS, feature toggles, language, auto_max_volume).

Contacts: quick_contacts.json

context.filesDir/quick_contacts.json

Managed by QuickContactRepository. Contains the full quick contact objects including photoUri and drawableResName (metadata not stored in config.json).

In-Memory: StateFlow

AppConfig.configFlow: StateFlow<ConfigData>

The reactive in-memory store. Compose UI collects this flow to recompose when config changes. Also accessible synchronously via AppConfig.current.

Sync Cascade on Save

Every call to AppConfig.save(context, config) triggers this cascade:

save(context, config)

1. _configFlow.value = config → UI recomposes

2. writeFile(config.json) → Persisted to disk

3. syncToSharedPrefs() → Legacy prefs updated

4. syncContactsToRepo() → QuickContactRepository merged

6. Configuration Editing (UI)

Users can modify configuration through three different interfaces:

A. Setup Wizard (First Run)

ui/pages/SetupWizardScreen.kt

A 7-page guided wizard that runs on first launch (tracked by SetupWizardPreference). Collects all essential settings step by step:

  1. Welcome — Introduction to the app
  2. Nickname — Configure the user's name for TTS
  3. Quick Contacts — Add favorite contacts with photo picker
  4. Language — Language selection (Romanian)
  5. Emergency Contacts — Add contacts for inactivity alerts
  6. Features & Theme — Theme selection, feature toggles, volume, inactivity monitor
  7. Settings Info — Instructions for accessing settings later and remote config

On completion, the wizard builds a ConfigData from all wizard state, calls AppConfig.save(), reschedules the inactivity monitor, and marks setup as completed.

B. Settings Screen (QuickContactSettingsScreen)

ui/pages/QuickContactEditor.kt

The main settings screen, accessed via a secret gesture (60 taps in 15 seconds on the home screen). Contains collapsible sections for every config aspect:

  • Quick Contacts (add/edit/delete/reorder with photo picker)
  • Apps (launch WhatsApp, Play Store, device settings)
  • TTS engine and toggle
  • Nickname editing with TTS preview
  • Language selection
  • Emergency contacts management
  • Theme selection (dropdown)
  • Feature toggles (all 6 features + auto max volume + inactivity monitor)
  • Permissions verification (re-run onboarding)
  • Advanced Config (JSON editor + URL import)

C. JSON Config Editor

ui/composables/JsonConfigEditorScreen.kt

A developer-oriented JSON editor with code-editor styling, accessible from the Advanced Config section of Settings. Features:

  • Fields grouped by ConfigGroup (General, TTS, Features, Behavior)
  • Syntax highlighting: booleans, strings, numbers
  • Type-specific editing: toggles for booleans, dropdowns for enums, text fields for strings/numbers
  • Info tooltips for each field (from ALL_CONFIG_FIELDS metadata)
  • Inline editing of quick contacts and emergency contacts arrays
  • "Reset to Defaults" and "Save" buttons at the bottom

7. Remote Import & Auto-Sync (URL)

AppConfig.kt:66-78, ConfigUrlPreference.kt, QuickContactEditor.kt (ImportUrlDialog), MainActivity.kt

The app supports importing configuration from a remote URL. This allows a caretaker to host a JSON config file (e.g., on a web server or cloud storage) and apply it to the device. Once a URL is set, the app automatically re-fetches the config every time the user returns to the main screen.

URL Storage (ConfigUrlPreference)

data/ConfigUrlPreference.kt

The config URL is stored independently from ConfigData in SharedPreferences (key config_url in LauncherPrefs), managed by the ConfigUrlPreference helper object. This is intentional — it prevents a remote JSON payload from overwriting or clearing the URL, which would break the auto-sync loop.

Design decision: The config URL is not part of ConfigData or config.json. It lives only in SharedPreferences, set exclusively from the Settings screen. Remote JSON imports cannot affect it.

Manual Import Flow (Settings)

User enters URL in ImportUrlDialog → AppConfig.importFromUrl()

↓
URL(url).readText() → JSONObject(text) → ConfigData.fromJson(json) → save(context, config)

↓
ConfigUrlPreference.setUrl(url) ← URL persisted for auto-sync

When the user imports via the dialog, two things happen: (1) the fetched config is applied immediately, and (2) the URL is saved via ConfigUrlPreference.setUrl() so it persists for auto-sync. The dialog also pre-fills with the previously saved URL when re-opened.

Automatic Sync Flow (onResume)

User returns to main screen → MainActivity.onResume() → fetchRemoteConfigIfNeeded()

↓
ConfigUrlPreference.getUrl() → URL blank? → skip

URL present ↓
CoroutineScope(IO).launch → AppConfig.importFromUrl() → Config updated silently

Every time onResume() fires in MainActivity, fetchRemoteConfigIfNeeded() checks the stored URL. If non-blank, it launches a background coroutine to fetch and apply the remote config. Errors are logged silently without disrupting the user.

Implementation details:

// In MainActivity
private fun fetchRemoteConfigIfNeeded() {
    val url = ConfigUrlPreference.getUrl(this)
    if (url.isBlank()) return
    CoroutineScope(Dispatchers.IO).launch {
        AppConfig.importFromUrl(this@MainActivity, url)
    }
}
Note: importFromUrl() uses java.net.URL.readText() which performs blocking network I/O. Both the manual import (via coroutine in ImportUrlDialog) and the auto-sync (via Dispatchers.IO in MainActivity) run this off the main thread.

8. Configuration Consumers

Config values are consumed throughout the app in several key areas:

Theme System

ui/theme/Theme.kt

val config by AppConfig.configFlow.collectAsState()
val themeMode = try { AppThemeMode.valueOf(config.themeMode) } catch (_) { AppThemeMode.SYSTEM }
// Selects dark/light/dynamic color scheme based on themeMode

The LauncherTheme composable observes configFlow reactively. Any change to themeMode triggers an immediate recomposition of the entire UI theme.

MainActivity (Home Screen)

MainActivity.kt

  • AppConfig.init(this) — first call in onCreate()
  • Uses NicknamePreference for TTS announcements ("Buna ziua, Mamaie!")
  • Uses FeatureTogglePreference to conditionally show/hide UI buttons (dial pad, contacts, messages, gallery, WhatsApp)
  • Reads quick contacts from QuickContactRepository (synced from config)

InactivityMonitorWorker

services/InactivityMonitorWorker.kt

  • Reads config.inactivityMonitorEnabled to decide whether to schedule
  • Reads config.inactivityMonitorThresholdHours for the alert trigger time
  • Reads config.emergencyContacts to know who to SMS
  • Reads config.userNickname for the SMS alert message text

ImmersiveActivity (Base Class)

All activities inherit from ImmersiveActivity which uses config values for:

  • TTS speech rate from ttsSpeechRate
  • Inactivity timeout from inactivityTimeoutMs
  • Speaker delay from callSpeakerDelayMs
  • Max missed call announcements from maxMissedCallAnnouncements

11. Data Flow Diagrams

Complete Configuration Lifecycle

LOADING (app start):
  config.json → JSONObject → ConfigData.fromJson() → _configFlow.value
                                                      → syncToSharedPrefs()

EDITING (user action):
  UI change → new ConfigData → AppConfig.save()
                                  → _configFlow.value = config   (UI recomposes)
                                  → writeFile(config.json)       (disk persist)
                                  → syncToSharedPrefs()          (legacy sync)
                                  → syncContactsToRepo()         (contacts merge)

REMOTE IMPORT (manual or auto-sync on every onResume):
  ConfigUrlPreference.getUrl() → HTTP GET → JSON string → JSONObject → ConfigData.fromJson() → save()
  (URL stored in SharedPreferences, independent of ConfigData — remote JSON cannot overwrite it)

CONSUMING:
  Compose UI     ←  collectAsState(configFlow)     ←  _configFlow
  Workers        ←  AppConfig.current              ←  _configFlow.value
  Legacy code   ←  SharedPreferences             ←  syncToSharedPrefs()
